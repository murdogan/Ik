## Discovery decision

P4A will evolve the existing `/api/v1/employees` compatibility surface instead of adding a
parallel employee API. Plain JSON success bodies, `X-Next-Cursor`, idempotent create, legacy
department/position fields, tenant guards and Phase 3 effective-dated assignment history remain
intact.

The smallest coherent contract is:

- add normalized tenant-scoped employee-number and optional work-email uniqueness, plus a minimal
  positive employee version column; null work email is allowed, while blank input is rejected;
- extend the bounded employee query with name search and current structured assignment ID filters,
  using one effective-assignment predicate and a batched projection rather than per-row loads;
- return an additive current-assignment summary while preserving legacy department/position output;
- audit employee create and actual update/archive changes in the command transaction with only
  allowlisted field names;
- add a permission-gated `/employees` Next.js directory, minimal create form and route-backed
  employee summary. Navigation and direct-page access require `employee:read:tenant`; create also
  requires `employee:update:tenant`.

Optional assignment inputs are deliberately not part of employee create. The existing employee
and assignment commands own separate transactions, so chaining them would permit a partially
created product state. HR can add the retained structured assignment through the existing
organization workspace. The permission catalog is not broadened: in particular, `tenant_admin`
continues to have no implicit employee read/update access.

## Implementation record

- `0032_p4a_employee_directory` adds generated, tenant-scoped normalized employee-number and
  nullable work-email keys, positive optimistic versions, immutable
  `(tenant_id, id)` directory and `(tenant_id, status, id)` status cursor indexes, and the two
  missing effective-assignment filter indexes. Unicode whitespace handled by
  Python `strip()` is also trimmed by the generated database expressions. Existing raw fields and
  the Phase 3 assignment/history tables are retained.
- The migration refuses normalized collisions or blank identifiers before mutation. Its
  downgrade refuses to erase any employee version that has advanced beyond `1`.
- The compatible employee API keeps plain object/list success bodies and `X-Next-Cursor`. It adds
  full-name search, four current-assignment UUID filters, `version`, and a compact structured
  assignment summary. Number/email prechecks and named database constraints map to deterministic
  `409 employee_number_conflict` and `409 employee_work_email_conflict` responses.
- Employee create, actual patch changes, and the first archive append field-name-only audit
  events in the command transaction. Idempotent create replay does not duplicate the event.
- `/employees` is permission-gated in both desktop/mobile navigation and direct routing. The HR
  flow includes directory loading/empty/error/filter/pagination states, a minimal create dialog,
  and a route-backed accessible summary dialog for the newly created employee.

## P4A review-blocker repair and scope reassessment

The final cross-dialect employee cursor payload contains only `id: UUID`. The service orders by
`Employee.id ASC` and its continuation predicate is exactly `Employee.id > cursor.id`; there is no
datetime normalization or dialect-specific `julianday` branch. The strict opaque resource schema
forbids extra fields, so both old `employee_number` and intermediate `created_at` payloads are
rejected. This keeps the plain-array body, `X-Next-Cursor`, `limit + 1`, filters, tenant predicate
and deprecated offset compatibility unchanged while preventing employee-number edits from
skipping an unseen row or duplicating an already-seen row across pages.

The independent review found that SQLite `julianday(created_at)` lost microsecond precision,
offset-aware cursor values were not normalized to the same instant before SQLite binding, and the
datetime expression ordering could not use the physical directory index. Because unpublished
P4A migration `0032` can still be edited in place, its cursor indexes are now `(tenant_id, id)` and
`(tenant_id, status, id)`. Regression coverage includes deterministic ID ordering, strict old-token
rejection, employee-number mutation between pages, and rows whose `created_at` values differ only
by microseconds; the latter no longer participates in cursor comparison.

The review's scope concern was reassessed against the approved P4A discovery contract above.
Optional optimistic `version` compatibility and field-name-only audit events for existing
create/update/archive commands remain because they are already additive P4A compatibility and
transaction-safety behavior; removing them in this blocker repair would be unrelated churn.
P4B/P4F still own richer employee profile and lifecycle capabilities. No such feature was opened.

Representative RED checks were observed before their minimal GREEN changes: normalized duplicate
number and email acceptance, missing model/version fields, stale-version 500 handling, ignored
structured filters and name search, missing audit arguments, missing migration head, unsafe
version downgrade, old smoke name-search behavior, and the absent `/employees` browser surface.

## Delivered vertical slice

### Persistence and database security

- Additive Alembic revision `0034_p4c_employee_account_links` creates one tenant-owned current-link
  table with UUID ID, tenant/employee/membership IDs, positive optimistic version and timestamps.
- Composite tenant FKs target `employees(tenant_id,id)` and
  `tenant_memberships(tenant_id,id)` with `ON DELETE RESTRICT`; tenant/employee and
  tenant/membership unique constraints enforce the one-to-one current link.
- The revision performs no `INSERT`, link backfill or email matching. Its downgrade refuses to drop
  non-empty current links.
- PostgreSQL enables and forces RLS, resets hostile inherited ACLs, grants the tenant app only
  `SELECT, INSERT, DELETE` plus column-scoped update of `membership_id`, `version`, `updated_at`,
  and grants no link-table access to platform/authentication/identity roles.
- A tenant-GUC-bound, pinned-search-path `SECURITY DEFINER` boolean function checks canonical
  membership, legacy compatibility user and global identity activity without exposing identity
  rows to the tenant application role.

### Backend and API

- Auth/session responses and `RequestContext` now carry the canonical `membership_id`; authenticated
  context construction rejects a membership mismatch even if tenant and actor happen to match.
- HR account-link endpoints are:
  - `GET /api/v1/employees/{employee_id}/account-link`
  - `GET /api/v1/employees/{employee_id}/account-link/eligible-memberships`
  - `PATCH /api/v1/employees/{employee_id}/account-link`
- The lookup is capped at 20 and returns canonical membership IDs only after same-tenant active
  membership/user/identity and permission-generation checks. Foreign, unavailable and already-used
  memberships share a safe conflict response.
- Strict PATCH requires both `membership_id` (nullable) and `expected_version` (nullable only for an
  initial link). Same-target and already-unlinked retries are idempotent; stale mutations return
  `409`; employee/link/membership locks plus DB constraints close races.
- `GET /api/v1/me/employee-profile` has no employee path parameter and resolves tenant, membership
  and actor only from the authenticated request context. Unknown employee query/header data is not
  consumed. Unlinked, archived, disabled/locked/invited/pending, permission-version-mismatched,
  legacy-actor-mismatched and cross-tenant states return the same ID-free unavailable projection or
  are denied by the existing live-session boundary.
- The own projection is separate from Employee 360 and contains only allowlisted current core,
  personal, employment and safe current-organization fields. It contains no assignment history,
  audit data, global identity ID, manager user ID, optimistic versions, tokens or secrets.
- Link/relink/unlink audit uses `employee.account_link.changed`, ID/status-only metadata and
  field-name-only changes. Service methods never commit; mutation plus audit runs in one request UoW,
  and forced audit failure rolls both link and unlink back.

### Frontend product

- Authorized HR receives an Employee 360 account-link card with current-link eligibility, bounded
  account search, canonical selection, explicit link/relink/unlink confirmation, optimistic conflict
  reload, and loading/empty/error/success states. The card is not mounted without update permission.
- Protected `/profile` is labeled `Profilim` in role-aware tenant navigation for
  `employee:read:own` personas. It calls only `/api/v1/me/employee-profile` and is read-only.
- Unlinked users receive an ID-free helpful state. A direct unauthorized route does not mount the
  own-profile client.
- Employee 360 and P4C clients bind responses to session generation, legacy actor, canonical
  membership, tenant, permission generation and permission; HR responses additionally bind the
  route employee. Wrong response identities and delayed old-session results are discarded.
- Tokens and refresh/session behavior remain unchanged: access tokens stay memory-only and refresh
  remains same-origin HttpOnly-cookie based.

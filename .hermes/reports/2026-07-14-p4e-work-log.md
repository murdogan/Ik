## Delivered design

### Typed persistence and request policy

`employee_profile_change_requests` is a tenant-owned, 25-column typed table with no JSON payload:

- Identity/ownership: `id`, `tenant_id`, `employee_id`, `requester_membership_id`,
  `requester_user_id`.
- Workflow: `status`, `version`, `base_profile_version`.
- Typed snapshots: `preferred_name_changed`, `previous_preferred_name`,
  `proposed_preferred_name`, `phone_changed`, `previous_phone`, `proposed_phone`,
  `birth_date_changed`, `previous_birth_date`, `proposed_birth_date`.
- Timing/decision: `submitted_at`, `decided_at`, `cancelled_at`,
  `decided_by_membership_id`, `decided_by_user_id`, `rejection_reason`.
- Row timing: `created_at`, `updated_at`.

Named checks enforce the closed `submitted`, `approved`, `rejected`, and `cancelled` states,
positive versions, at least one change, coherent typed snapshots, timestamp order, and terminal
attribution. Tenant-scoped FKs cover employee, requester/decider membership, and requester/decider
user attribution. A PostgreSQL partial unique index enforces one `submitted` row per
`(tenant_id, employee_id)`; queue and own-history indexes match deterministic cursor orders.

P4D's central registry now carries a deny-by-default `profile_change_requestable` flag. Its exact
true set is `personal.preferred_name`, `personal.phone`, and `personal.birth_date`; every core,
employment, organization, account-link, identity, version, and restricted-future entry remains
false. The closed create schema distinguishes omission from explicit null and accepts no employee,
user, membership, tenant, field-name, path, or model-dump target. It rejects unknown/forbidden
keys, empty commands, every selected normalized no-op, masked-display strings, blank/overlong or
control-character names, malformed/overlong phones (including unbalanced or empty parenthesis
groups), malformed dates, and non-strict expected versions such as booleans or strings. Rejection
reasons are bounded and reject control characters. All three existing personal fields are nullable,
so explicit clearing is supported for exactly those fields.

### Routes and permissions

| Surface | Route | Permission contract | Public shape |
| --- | --- | --- | --- |
| Employee | `POST /api/v1/me/profile-change-requests` | `employee:read:own` + live canonical link | own masked detail |
| Employee | `GET /api/v1/me/profile-change-requests` | same | bounded newest-first list |
| Employee | `GET /api/v1/me/profile-change-requests/{request_id}` | same + own requester/employee binding | own masked detail |
| Employee | `POST /api/v1/me/profile-change-requests/{request_id}/cancel` | same + expected version | cancelled own detail |
| HR | `GET /api/v1/employee-profile-change-requests` | both `employee:read:tenant` and `employee:update:tenant` | bounded filtered oldest-first summaries |
| HR | `GET /api/v1/employee-profile-change-requests/{request_id}` | both grants | typed base/current/proposed comparison |
| HR | `POST /api/v1/employee-profile-change-requests/{request_id}/approve` | both grants + expected version | approved comparison |
| HR | `POST /api/v1/employee-profile-change-requests/{request_id}/reject` | both grants + expected version/reason | rejected comparison |

Own routes never accept an employee ID. Their target is resolved only from authenticated tenant,
actor user, membership, and the live P4C canonical account link. Own phone/date snapshots are
masked through P4D and unavailable values stay explicit; no requester/decider membership or global
identity IDs are public. The own response includes the same employee ID already public on P4C
Profilim so the client can reject a mismatched response, but it is never an input target.

HR queue summaries contain request state/version/timestamps/field names, base/current versions,
stale flag, and safe employee `id`, number, name, email, and status. Raw typed values appear only on
the both-permission HR detail/decision surface. Managers with team permissions alone are denied.
Stable non-leaking errors are 422 `employee_profile_change_request_validation_error`, 404
`employee_profile_change_request_not_found`, 409 `employee_profile_change_request_conflict`, and
409 `employee_profile_change_request_stale_profile`.

Own history uses a bounded `(submitted_at, id)` newest-first cursor and two fixed queries (live
target plus page). The HR queue uses one bounded joined query and an oldest-first `(submitted_at,
id)` cursor scoped to the selected status. Neither path has an N+1 query.

### Transactions, PostgreSQL security, and audit

Submit resolves the live P4C link, checks for an active request before taking the profile lock,
then locks the personal profile and takes its typed version/value snapshot. The partial unique
index remains the final concurrency authority, so concurrent submits produce one `submitted` row
and one deterministic active-request conflict. Cancel/reject/approve lock the request and require
`submitted` plus the expected request version. Approval then locks the profile, requires its
current version and every selected base value to match, applies only the three typed proposed
columns, increments the profile version exactly once, and transitions the request to `approved` in
the same transaction. A stale profile returns the specific conflict with no partial write and
leaves the request at version 1 in `submitted`. Reject/cancel increment only the request version
and never mutate the profile. Race tests cover duplicate submit, approve/approve, approve/reject,
approve/cancel, and submit/approve with one coherent winner and no double profile increment.

Forward-only revision `0035_p4e_employee_change_requests` adds FORCE RLS and explicit tenant
policies. The tenant runtime receives only column-level SELECT on frozen request columns and no
direct INSERT/UPDATE/DELETE/TRUNCATE/REFERENCES/TRIGGER/DDL. The private `p4e_command` schema holds
a FORCE-RLS command-binding table keyed to backend PID and PostgreSQL transaction ID. Before the
runtime role is selected, the database gateway binds the exact intent, target, tenant, actor,
membership, audit ID, and correlation IDs; the command claims that capability once and leaves a
tombstone that prevents rebinding or reuse in the same transaction.

Six private-owner, fixed-search-path SECURITY DEFINER functions implement binding, one-shot claim,
value-free audit insertion, own submit, request transition, and the existing P4B HR
personal-profile update contract. Public command functions are executable only by the tenant app
capability role; the app role and PUBLIC cannot call the binder or private helpers. Live
membership/link and HR permission checks are repeated in SQL, and direct app UPDATE on personal
profile columns is revoked, so approval and P4B writes cannot bypass checked commands. Tests attack
forged tenant/actor/requester/status/decision context, direct helper access, rebinding, capability
reuse, cross-tenant IDs, and direct profile writes. Downgrade refuses retained request history and,
when empty, restores the P4B column grant without damaging profiles or links.

Successful submit, approve, reject, and cancel record explicit command-function-enforced,
same-transaction audit events. Metadata is limited to request/employee IDs, status transition,
changed field names, and a closed reason code. Old/new names, phones, dates, rejection free text,
membership/global identity IDs, tokens, and payloads are absent. Conflict attempts intentionally
do not persist an audit row: current repository convention rolls the failed command transaction
back rather than recording attempt events separately. No outbox event was added because P4G owns
notifications and no current approved P4E delivery pattern applies.

## Residual risks

- The browser journey uses the repository's deterministic route-mocking pattern; the isolated API
  smoke and real-PostgreSQL file separately prove the live server and database boundaries. P4H
  remains responsible for exhaustive integrated regression and performance coverage.
- Binder EXECUTE is granted to direct app-capability login members discovered at migration time;
  future gateway login provisioning must explicitly receive the same grant.
- Read isolation continues to use the repository-wide tenant-GUC gateway boundary. P4E's one-shot
  binding specifically hardens mutations; it binds intent and target rather than the full typed
  payload, so the gateway remains a trusted application boundary.
- PostgreSQL audit insertion is enforced by mutation command functions rather than a table trigger.
  The dedicated ACL/attack tests establish the supported write contract, but a future command must
  continue to call the private audit helper explicitly.
- Stale approvals remain submitted for explicit HR reload and rejection; no undocumented conflict
  status or automatic rebase was invented. A later employee request is possible after the stale
  request becomes terminal.
- Notifications/outbox delivery, encryption/reveal behavior, and P4F+ work are deliberately absent.

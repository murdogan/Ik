## Field ownership and API decision

- `employees` remains the compatibility owner for employee number, first name, last name, work
  email, status and employment start date. Existing GET/PATCH contracts remain additive-compatible.
- `employee_profiles` owns only preferred name, birth date and phone.
- `employee_employments` owns only contract type and work type. Its section presents and atomically
  updates the compatibility-owned employment start date.
- Both focused records are tenant-owned one-to-one records with independent positive optimistic
  versions. A section PATCH always requires its expected version; an employee expected version is
  additionally required when the command changes a compatibility-owned field.
- `GET /api/v1/employees/{employee_id}/profile` is the Employee 360 aggregate. It composes a bounded
  Phase 3 assignment projection and does not own or mutate assignment data.
- Personal and employment PATCH endpoints are separate atomic commands. Personal may change only
  first name, last name, work email, preferred name, birth date and phone. Employment may change
  only employment start date, contract type and work type.
- P4B audit events may carry before/after values only for the exact changed-field allowlist of their
  section. Existing P4A events remain field-name-only.

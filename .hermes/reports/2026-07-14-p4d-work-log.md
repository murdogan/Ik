## Central deny-by-default policy

The backend policy is one cohesive boundary split into three small modules:

- `employee_field_policy.py` is the closed field registry, classifications, per-audience
  visibility, and the only phone/birth-date masking implementation.
- `employee_field_projection.py` contains explicit constructors for HR, manager-team, and linked
  employee responses. It calls the central policy for every value and performs no wildcard/model
  copying.
- `employee_projection_contract.py` owns exact recursive response-leaf and ORM-column coverage.
  It rejects missing classifications, excluded fields, Pydantic computed fields, and custom model
  serializers that could otherwise add implicit output keys.

Current Employee, EmployeePersonalProfile, EmployeeEmploymentProfile, and EmployeeAssignment ORM
columns are covered exactly. The registered public employee contracts are EmployeeRead, P4B
EmployeeProfileRead and its mutation responses, the deprecated Phase 3 team list, the additive
manager-safe team list, the manager profile, and the P4C own state. A persisted or response field
addition therefore requires an explicit classification before focused coverage can pass.

The policy categories are:

- `work_safe`: work identity, status, employment relationship, and safe current organization
  display fields;
- `personal_contact`: phone, birth date, and the masked display payload carrying those values;
- `operational_internal`: tenant/link/assignment/version/history control metadata that is not a
  manager or own-profile field;
- `authenticated_session_boundary`: the single exact outer own-response `membership_id` leaf. It
  is full only for employee-own scope and excluded from HR and manager scopes; there is no wildcard
  membership allowance;
- `restricted_future`: policy-only sentinels for `bank_account`, `compensation`,
  `emergency_contact`, `health`, `iban`, `national_identifier`, `passport`, `payroll`,
  `private_address`, `special_category`, and `tckn`. They are not schema or persistence fields and
  have no audience.

## Exact field matrix

### Authorized HR tenant projection — unchanged P4B behavior

- `core`: `id`, `employee_number`, `first_name`, `last_name`, `email`, `status`,
  `employee_version`.
- `personal`: `preferred_name`, raw/null `birth_date`, raw/null `phone`, `version`.
- `employment`: `employment_start_date`, `contract_type`, `work_type`, `version`.
- `organization`: `history_limit`, `history_truncated`, and current/bounded-history assignment
  records containing:
  - assignment `id`, `effective_from`, `effective_to`, `supersedes_assignment_id`,
    `change_reason`, `is_current`, `created_at`, `updated_at`;
  - employee `id`, `employee_number`, `first_name`, `last_name`, `email`, `status`;
  - legal entity/branch/department `id`, `code`, `name`, `status`;
  - position `id`, `code`, `title`, `status`;
  - manager `id`, `full_name`, `email`, `status`, or null.

This projection remains protected by `employee:read:tenant`. Manager team permission does not
authorize it.

### Current direct-manager projection

`GET /api/v1/teams/me` remains the deprecated Phase 3 compatibility route. Its operation ID and
`TeamMemberRead` schema remain unchanged: outer `employee` is `AssignmentEmployeeRead` and
`assignment` is the complete `EmployeeAssignmentRead`. Because that legacy shape contains HR-level
operational fields, access requires both `employee:read:tenant` and `employee:read:team`. Team-only
managers receive `403`. The OpenAPI operation is `deprecated: true`, directs consumers to
`/api/v1/teams/me/members`, and records future removal after the deprecation window.

`GET /api/v1/teams/me/members` is the additive manager-safe list and returns:

- `employee`: `id`, `employee_number`, `first_name`, `last_name`, `preferred_name`, `email`,
  `status`;
- `assignment`: legal entity/branch/department `{code,name}`, position `{code,title}`, and
  `effective_from`.

`GET /api/v1/teams/me/members/{employee_id}/profile` returns:

- `core`: the same seven safe work-identity fields;
- `employment`: `employment_start_date`, `contract_type`, `work_type`;
- `organization.current_assignment`: legal entity/branch/department `{code,name}`, position
  `{code,title}`, manager `{full_name}` or null, and `effective_from`.

The manager-safe list and detail responses structurally omit phone, birth date, profile/employee
versions, tenant, membership/global/legacy identity IDs, assignment and organization UUIDs,
manager user ID/email/
status, account-link state, assignment history, history controls, supersession/change reason,
timestamps, audit payloads, and every restricted-future sentinel.

### Linked employee own projection

`GET /api/v1/me/employee-profile` remains employee-ID-free as an input and resolves only the live
RequestContext membership/actor link. Its response is:

- outer state: `availability`, P4C-compatible authenticated-session `membership_id`, additive linked
  `employee_id`, and `profile`; both IDs and profile are null for the uniform unavailable state;
- `core`: `id`, `employee_number`, `first_name`, `last_name`, `email`, `status`;
- `personal`: `preferred_name`, masked-state `birth_date`, masked-state `phone`;
- `employment`: `employment_start_date`, `contract_type`, `work_type`;
- `organization.current_assignment`: legal entity/branch/department `{code,name}`, position
  `{code,title}`, and manager `{full_name}` or null.

For a present value, birth date displays `••••-MM-DD` and phone displays eight bullets plus the
last two numeric digits. A null source becomes `{visibility:"unavailable",display_value:null}`.
There is no unmask endpoint, input, permission, button, flag, or placeholder. Available responses
require `membership_id` and `employee_id`, and require `employee_id == profile.core.id`.
`membership_id` appears only at the outer own-session response boundary; it is absent from the
nested profile, manager/HR projections, own-profile logs, and errors. Own-profile reads add no
audit event or payload; existing P4C account-link mutation audit remains unchanged and ID-only.
Own responses omit global/legacy identity IDs, manager user ID, all optimistic versions, history,
audit, and restricted-future fields.

## Authorization, query, audit, and error boundaries

- Both manager list routes derive tenant and manager actor exclusively from RequestContext and use
  the same current, non-archived, active/on-leave direct-assignment predicate. Neither accepts a
  client manager, department, or scope input. The legacy route adds tenant-read to team-read and
  fails closed before data access: the API requires tenant-read, and the service independently
  requires both tenant-read and team-read.
- The candidate employee UUID on the manager detail route identifies a resource only. Tenant and
  manager actor come exclusively from RequestContext. The one data query requires tenant match,
  `manager_user_id == actor_id`, `[effective_from,effective_to)` current effectiveness, a
  non-archived employee, and active/on-leave status.
- Unrelated same-tenant, former, future, archived, indirect, cross-tenant, and guessed UUIDs all
  use the same generic assignment-not-found response. Client-provided manager/department/scope
  query or header values are ignored.
- The manager profile is one explicit-column joined `LIMIT 1` query after the two constant
  tenant/feature gates. A fixture with 55 history rows still executes exactly three SELECTs and
  emits no history; selected SQL contains neither phone nor birth date.
- Existing live session, membership, actor, tenant, token-family, and permission-generation checks
  are unchanged and run before the new endpoint.
- No employee-read audit event exists or was added. Manager successes and denials write zero audit
  events. Completion logs and generic errors receive no profile/query/header values. Focused tests
  inject phone, birth date, and a unique value for every restricted-future name and prove none is
  present in manager responses, denial envelopes, completion records, or audit state.

## Frontend product

- Dashboard `Ekibim` consumes only `/teams/me/members` and links each current direct report to
  `/team/{employeeId}`. The focused browser journey treats any legacy `/teams/me` request as a
  failure and proves the count remains zero.
- The `/team` layout checks `employee:read:team` before mounting the data client. The responsive,
  accessible detail screen calls only the manager-safe endpoint and includes loading, empty,
  retry/error, and safe work/organization states.
- Both manager list and detail clients validate exact response keys. Detail rejects a response
  whose `core.id` differs from the route employee. Requests bind session generation, user,
  membership, tenant, permission generation, permission, and route employee; delayed successes,
  failures, and pagination from an old boundary are ignored.
- Employee 360 behavior is unchanged. `Profilim` consumes the centralized masked own contract,
  explains masked/unavailable values, and contains no reveal control. Its client rejects extra
  keys, requires outer `membership_id` to equal the captured current session membership, and
  requires outer `employee_id` to equal `profile.core.id`. Existing session-generation guards
  discard delayed results from stale user/membership/tenant/permission boundaries.
- Proxy coverage now includes `/team/:path*`; role-aware layouts and navigation prevent
  unauthorized data-client mounting.
